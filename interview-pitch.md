## スライドごとのスピーカーノート（10分詳細版）

### S1: 表紙

**話す内容（30秒）**

> 多段Proxy・認証・ログ基盤を OSSで自分の手で組み直した検証成果物です。
> 主役はオンプレ相当検証環境で、実装・検証まで完了しています。AWS への移行案も整理していますが、AWS 上の実機構築は未実施で、設計のみの段階です。

### S2: OVERVIEW（状態の地図）

**話す内容（1分）**

> このスライドで最初に状態を分けて報告します。
> 実装済はオンプレ相当検証環境、検証済は稼働確認・PAC経路・ログ切り分け・再構築の4観点、設計済は AWS 移行設計案、そして AWS 実機構築は未実装です。
> 「動かした」と「設計した」と「未実施」を混ぜずに伝えるための地図として置いています。

### S3: BACKGROUND（なぜ作ったか）

**話す内容（1分）**

> 実務では障害解析がベンダー回答に依存しやすく、設計意図や原因を自分の言葉で説明しづらい場面がありました。
> そこで、入口から出口まで自分で構築し、通信・認証・暗号化・ログを横断してログ根拠で説明できる状態を作る、という目標で組み直しました。

### S4: 全体構成図（ONE DIAGRAM 詳細版）

**見せる**: 中央に Client → Proxy1 → Proxy2 → Proxy3 → Internet。下段に Samba AD/DC（VMware）/ dnsmasq / stunnel / ICAP+ClamAV / Loki / Graylog / OpenSearch / Zabbix。

**話す内容（1分30秒）**

> WSL2 + Docker Compose を中心に、VMware上の Samba AD/DC・dnsmasq と連携した構成です。
> 中央が3段Proxy、下段が認証・DNS・暗号化・検査・ログ・監視を横断機能として支えるレイヤです。
> 「すべて Docker で完結」ではなく、AD/DC と dnsmasq は VMware 側に置いて実務に近い前提を保っています。

### S5: 3段Proxyの責務分離

**話す内容（1分30秒）**

> Proxy1 が入口・認証・復号点、Proxy2 が分岐・ACL、Proxy3 が出口です。
> SSLBump は1通信につき1回しか成立しない仕様で、当初 Proxy1 と Proxy2 の両方で復号して破綻しました。
> 失敗ログから把握して、復号点は Proxy1 のみに集約し、中継暗号化は stunnel に分離する責務分離へ変更しています。

### S6: 認証・DNS・PAC

**話す内容（1分）**

> 認証は Samba AD/DC + Kerberos、名前解決と PAC 配布は dnsmasq が担います。
> PAC 分岐は「実際に経路差が生じたか」をログで裏取りしており、Proxy 経由のときは Proxy1 のログに残り、DIRECT のときは残らない、という判定が立つ状態を作っています。

### S7: 横断切り分け Evidence

**話す内容（1分30秒）**

> 切り分けは access.log から始めます。
> Case A は Proxy1 で 407、Proxy2 では No logs found。入口で止まっているのが結果ログから分かります。
> Case B は Proxy2 で 403、cache.log を見ると ACL 拒否が確定します。
> 「結果ログで発生段を特定し、必要に応じて原因ログへ寄せる」流れを、2ケースで言い切れる状態にしています。
> ICAP は補足として位置づけ、本ピッチでは深追いしません。

### S8: ログ・監視基盤

**話す内容（1分）**

> ログの流れは「結果 → 原因 → 監視」です。
> access.log で結果（HTTP ステータス・経路）を確認し、cache.log で原因（TLS / ACL / 上流不達）を絞り込みます。
> 横展開として、Loki は一次切り分け（ラベルでの絞り込み）、Graylog / OpenSearch は全文検索、Zabbix は継続監視という役割分担にしています。

### S9: 再現性 Evidence

**話す内容（1分）**

> 「一度動いた」ではなく、何度でも同じ状態へ到達できることを重視しました。
> 環境破棄から、コンテナ起動だけでなく、3段Proxy・認証・DNS/PAC・ログ収集・監視・通信制御まで確認できる状態へ、約25分で戻せます。
> 実測は STEP0-17 完走で 22分51秒。失敗しやすい箇所は Guard / Retry / Warn 継続で吸収し、最後に総所要時間を必ず出力する設計です。

### S10: AWS移行設計案

**話す内容（1分）**

> 次のフェーズとして、同じ責務分離を AWS のマネージドに写像する設計案を整理しました。
> Phase 1 は 1AZ の最小構成、Phase 2 で 2AZ の標準化、Phase 3 で OpenSearch 等の拡張、と段階的に組み立てています。
> 認証は AD Connector、ログは CloudWatch Logs と S3、運用は SSM、到達経路は Client VPN という対応です。
> AWS 上の実機構築は未実施で、次の一手として Phase 1 を Terraform で構築し、CloudWatch Logs / SSM / S3 を実機検証する予定です。

### S11: CLOSING

**話す内容（30秒）**

> 実装・検証済みの到達点は、3段Proxy・認証・DNS・暗号化・検査・ログを束ね、約25分で検証可能状態まで再構築できることです。
> 示せる能力は、ログ根拠で経路を切り分けること、責務分離で設計判断ができること、再現性ある自動化を組めることの3点。
> 設計済（未実装）は AWS 移行設計案。次の一手は Phase 1 を Terraform で実機検証する、という流れで締めます。

## 質疑想定（簡易版）

| 想定質問 | 用意している答えの軸 |
|---|---|
| 一番苦労したところは？ | 多段 SSLBump が成立しない仕様制約。失敗ログから把握し、復号点を Proxy1 に集約・中継暗号化を stunnel に分離して責務分離した経緯を話す |
| 商用設定の真似ではない？ | 構造と責務から自作。実環境のコピーは行っていない。機密情報は一切含まない |
| 規模感はどれくらい？ | コンテナ約20 / STEP0-17 のスクリプト群 / 検証時間 約25分（実測 22:51） |
| AWS 経験はどの程度？ | 設計案と判断軸を整理した段階。実機構築は次フェーズで Terraform / CloudWatch Logs / SSM / S3 を実機検証する予定 |
| 運用面の工夫は？ | STEP 番号での順序固定、Guard / Retry / Warn 継続、所要時間の必須出力で「どこで止まったかを切り分けやすくする」ことを優先 |
| なぜポートフォリオを作った？ | 設計意図や原因を自分の言葉で説明しづらい状況を変え、ログ根拠で切り分けて説明できる状態を作るため |

## 補足: AWS 移行設計案だけを掘り下げて説明する場合

AWS 移行設計案そのものを 5 分程度で掘り下げて話す場合のメモです。詳細は [AWS 移行設計案](./aws-deployment-plan.html) を参照してください。

> **注意（番号表記）**: AWS 図面では `Proxy1=出口 / Proxy2=中継 / Proxy3=入口・認証` の番号を採用しています（OSS と反転）。詳細は [AWS 移行設計案 §0-4](./aws-deployment-plan.html) を参照。

| # | 話す内容 | 所要時間 | 着地点 |
|---|---|---|---|
| 1 | 本ページは「設計案」「AWS 上の構築は未実施・次フェーズで検証予定」 | 30秒 | 実装段階ではなく **設計判断を整理した段階** であることを明示 |
| 2 | 構成比較図 → 案A（単純移植）/ 案C（監査強化）と比較した B 案の採用根拠 | 1分 | コスト・可用性・運用負荷の判断軸 |
| 3 | 全体構成図 → 3段Proxy（Proxy3→Proxy2→Proxy1）で OSS の責務分離を AWS でも維持 | 1分 | VPC / 2AZ / NAT / CW / AD Connector + Samba AD/DC / PAC を一枚図で説明 |
| 4 | レイヤー分解図 → Client / Access / Control / Auth / Proxy / Inspection / Egress / Logging の 7 レイヤ | 45秒 | レイヤ別責務分離と Logging Layer の横断観測 |
| 5 | Phase 1 → 2 → 3 の段階導入と「最初から全部入れない」理由 | 45秒 | 検証コスト・運用負荷・障害切り分けの観点 |
| 6 | 想定論点を 1 つ Q&A 形式で（例：AD Connector + Samba AD/DC で何を検証するか） | 45秒 | DNS / Kerberos / NTP / SG / 逆引きの 5 観点 |
| 7 | 承認付き IaC 展開フロー | 30秒 | 自動化＝楽ではなく、レビュー可能で再現性の高い展開 |

### 30秒で言い切る要約

> 既存 OSS 3 段 Proxy で分離した「通信・認証・暗号化・ログ・監視」の責務を、AWS 上でも 3 段 Proxy + 2AZ + NAT Gateway 2 台の標準構成で維持しつつ、可用性・セキュリティ・コスト・運用性の観点で再設計した設計案です。
> Phase 1（最小）→ Phase 2（B 案標準）→ Phase 3（OpenSearch / Network Firewall / Managed AD など将来拡張）の段階導入で、過剰構成を避ける設計判断とその理由まで明記しています。
> 顧客アカウントへの展開は、最小権限・承認・監査・実行後検証を含む承認付き IaC 展開フローとして整理しています。AWS 上の実機構築は未実施で、次フェーズで Terraform による実機検証を予定しています。
